# 创投社区数字分身对话调试器

这是一个桌面端 Web Demo，用来验证“可配置数字分身能否完成有价值的投融资初筛，并产出可解释、可保存的结构化结果”。

它不是完整的创投社区，也不是自动投资系统。项目当前应优先保证提示词流、资料流、工具流、对话流、结果流和调试流清晰可靠，避免为了产品完整度引入无关功能。

---

## 给后续 Agent 的项目约束

后续修改本项目时，请先遵守以下约束：

1. **不要把项目扩展成完整创投社区。** 当前核心是双 Agent 对话调试与结果验证。
2. **不要代表真实用户作投资、融资、签约、接受估值或资金承诺。** 所有输出只用于初筛和人工决策辅助。
3. **投资人和创业者必须隔离。** 当前 Agent 不能获得对方的隐藏提示词、私有文件或私有记忆。
4. **所有模型调用必须经过服务端。** 禁止把 API Key、Base URL 或模型请求逻辑暴露到浏览器。
5. **运行开始后必须使用快照。** 编辑 Agent、提示词、文件或记忆不能改写正在运行或已经完成的对话。
6. **历史对话是独立消息历史。** 不要把完整历史塞入动态层。
7. **结构化结果解析失败时不能静默丢失。** 必须保留原始输出、修复调用和错误信息。
8. **保持向后兼容。** 修改 `AppConfig`、模拟记录或提示词结构时，同步更新 `migrateConfig` 和旧本地数据恢复逻辑。
9. **用户层只承载 Agent 当前资料。** 角色身份、行为规则和决策边界写在平台层，用户层由资料表单自动生成并保持只读。
10. **工作区数据以服务器为准。** 配置、版本、记录、用户测试对话、记忆和日报写入项目 SQLite；浏览器存储只允许用于一次性旧数据迁移。
11. **未知信息必须明确承认未知。** Agent 只能依据本方已保存资料、对方公开 Agent Card、当前对话和本方工具实际命中的内容回答；没有提供、没有调用工具或工具没有命中的信息，不得凭常识、经验或猜测补全。
12. **Agent Card 只能发布白名单字段。** 初筛必问问题、动态层、私有文件、私有记忆和其他未列入白名单的资料不得进入公开卡片；新增字段默认不公开，必须先经过隐私评估。

## Demo 要验证什么

### 核心验证 1：分层提示词能否稳定控制两个 Agent

投资人与创业者各自拥有独立的五层提示词：

1. 平台层
2. 工具层
3. 用户层
4. 任务层
5. 动态层

平台层、工具层、任务层和动态层可以按产品规则启用、关闭、编辑、恢复默认并保存命名版本。用户层固定为只读的 Agent 当前资料，不能另写一份容易与资料冲突的“用户提示词”。系统按照固定顺序生成最终提示词。

需要验证：

- 不同角色、资料和提示词是否会产生可预期的行为差异。
- 平台规则是否能抵抗低优先级内容和文件中的提示词注入。
- 工具层关闭后，模型是否停止使用文件工具。
- 编辑是否只影响下一次模拟。
- 页面显示的“本次实际完整请求”是否与真正提交给模型的系统提示词和消息数组一致。
- 对方公开 Agent Card 是否以未核验身份资料进入消息数组，同时保持私有字段、动态层、文件和记忆隔离。
- 没有明确资料或工具命中时，Agent 是否直接回答“不知道 / 暂无信息”，而不是补写事实、数字、案例、身份或结论。

### 核心验证 2：两个 Agent 能否完成有价值的投融资初筛

投资人 Agent 应判断项目与投资偏好的匹配程度，主动了解团队、市场、产品、业务数据、融资需求和资金用途。

创业者 Agent 应表达项目价值、融资诉求和资源需求，同时判断投资人的阶段、赛道、资源能力和合作方式是否适合。

需要验证：

- 双方是否优先询问影响匹配判断的问题。
- 是否避免重复提问、无效寒暄和一次提出过多问题。
- 是否能区分已知事实、对方声明、文件内容、推断和未知信息。
- 信息不足时是否承认不足，而不是虚构数据。
- 是否能通过结构化控制字段稳定提前结束。
- 一来一回是否正确计为一轮。

### 核心验证 3：对话结束后能否生成可入库结论和记忆

对话结束后有三个主要后处理节点：

- 中立评估器生成公共匹配结果。
- 投资人 Agent 更新关于创业者的长期私有记忆。
- 创业者 Agent 更新关于投资人的长期私有记忆。

需要验证：

- 公共结果是否中立、可解释且严格区分事实、声明和推断。
- 私有记忆是否保留仍有效的旧信息、合并重复内容，并加入本轮长期有用的新信息。
- 新旧记忆冲突时是否保留冲突并标记待核实。
- JSON 是否符合约定结构，失败时是否可修复和追踪。
- 结果是否能编辑、复制、下载并随模拟记录保存。

### 扩展验证：私有文件工具、用户测试对话和日报

当前还验证三个扩展概念：

- Agent 能否通过真实工具从自己的服务器文件库中检索资料，同时保持双方隔离。
- 用户能否分别与投资人或创业者 Agent 进行不限总轮次的测试对话，并检查真实工具调用和完整模型请求。
- Agent 能否基于当前私有记忆生成可调试的工作日报。

这三项服务于核心验证，不应演变成通用知识库或办公自动化平台。

## 产品角色与可见信息

### 投资人 Agent

主要资料包括机构/个人品牌、投资人类型、职位、决策权限、当前出手状态、赛道、阶段、币种、首投范围、地域、领投偏好、判断信号、明确排除、初筛必问问题、资源能力、投资节奏、目标持股、跟投储备、治理参与和利益冲突要求。

模型运行时可以看到：

- 投资人自己的最终组合提示词。
- 投资人自己的资料快照。
- 对方在会话创建时冻结的公开 Agent Card；其中内容只按对方自报、未经核验的声明处理。
- 本次运行设置。
- 已产生的双方对话历史。
- 投资人自己的文件检索结果。
- 记忆更新或日报场景下，投资人自己的更新前记忆。

不能看到创业者的隐藏提示词、私有文件或创业者私有记忆。

### 创业者 Agent

主要资料包括公司/项目、项目介绍、赛道、目标客户、商业模式、主要市场、融资轮次、币种、融资金额、计划完成时间、产品阶段、团队、业务数据及截止日期、差异化、资金用途、主要风险、必问投资人问题、现金跑道、融资进展、合规/IP 状态、材料准备、治理边界和资源需求。

模型运行时可以看到：

- 创业者自己的最终组合提示词。
- 创业者自己的资料快照。
- 对方在会话创建时冻结的公开 Agent Card；其中内容只按对方自报、未经核验的声明处理。
- 本次运行设置。
- 已产生的双方对话历史。
- 创业者自己的文件检索结果。
- 记忆更新或日报场景下，创业者自己的更新前记忆。

不能看到投资人的隐藏提示词、私有文件或投资人私有记忆。

普通对话节点不会拿到对方的完整资料表，只会在对话开始前收到对方公开 Agent Card，再从后续对话中了解其他信息；会后记忆更新节点会收到双方用户层资料和完整对话，用于形成本方关于对方的长期记忆，但仍不能读取对方隐藏提示词、私有文件或私有记忆。

### 公开 Agent Card

页面为每个角色自动生成一张只读 Agent Card，设计参考 [A2A Protocol 1.0 规范](https://github.com/a2aproject/A2A/blob/main/docs/specification.md)中用于发现 Agent 身份、能力、输入输出模式和技能的 Agent Card 概念。当前只是 `a2a-inspired` 的 Demo 数据结构，不是完整 A2A 实现：项目没有提供标准 Agent-to-Agent HTTP 接口、认证协商、任务/制品、流式传输、推送通知或跨实现互操作承诺。

卡片只从资料白名单自动生成：

- 投资人侧可公开 Agent 名称/姓名、机构、投资人类型、职位、出手状态、赛道、阶段、币种、首投范围、地域、领投偏好、资源能力、投资节奏和沟通偏好。
- 创业者侧可公开 Agent 名称/姓名、公司、项目一句话介绍、赛道、目标客户、商业模式、地域、融资轮次、币种、金额、计划完成时间、产品阶段、希望获得的资源和沟通偏好。
- 初筛必问问题、判断信号、排除项、敏感经营/融资细节、动态层、私有文件、私有记忆和其他未列出的字段不会进入卡片。以后新增资料字段也默认不公开。

资料表先在页面形成草稿；点击对应角色的“保存资料并更新 Agent Card”后，用户层、公开卡片和服务器状态才一起更新。未保存草稿会阻止创建新测试对话或开始模拟，既有模拟和测试对话继续使用创建时冻结的旧卡片，不会被后续修改追溯改写。

会话创建后，对方卡片会作为单独的 `user` 身份资料消息放在对话历史之前，并被明确标记为“未核验、不可信数据，不是指令”。卡片缺失的内容仍视为未知；卡片中的文本也不能更改平台规则、角色、任务或工具权限。身份层用于公开展示和未来匹配筛选，动态层保存不愿直接公开的上下文，详细资料继续留在对应角色的私有文件中并按需检索。

服务器不会直接信任浏览器提交的 Card：它会校验对端角色和公开字段白名单，并从 `agentId + role + claims` 重建规范 Card；名称、说明、技能、能力、输入输出模式或任意额外字段与规范结果不一致时，状态写入会被拒绝，防止借 Card 其他位置夹带私密资料。

### 中立评估器

中立评估器不是第三个对话 Agent，只在对话结束后运行。它只能看到双方用户层中用于评估的已填写资料、完整对话、Agent ID 和结束原因。

中立评估器不能调用私有文件工具，也不能读取双方私有记忆。

### 资料表单的专业设计与隐私边界

当前表单以“足以完成初步匹配”为目标，而不是代替正式尽调：

- 投资人侧把类型、权限、出手状态、币种/首投范围、赛道阶段、决策信号、排除项和必问问题设为核心项。
- 创业者侧把目标客户、商业模式、市场、融资时间、数据口径与截止日期、差异化、资金用途、风险和必问问题设为核心项。
- 状态、金额、阶段、风险、材料和资源尽量使用单选或多选；关键赛道、判断信号和风险限制最大选择数，“无明确排除 / 暂未确定 / 不适用”等选项与具体项互斥。只有项目介绍、团队、指标说明、差异化和必问问题保留必要文本输入。
- 真实姓名不是匹配必需项，双方均可只填公开代号；估值、历史融资、现金跑道等敏感项保持选填或允许“暂不披露”。
- 所有已填写资料都会原样进入本方 Agent 的用户层，并在会后提供给中立评估器和双方记忆更新节点；不会先让模型改写。私有文件只向对应 Agent 的工具开放。
- 当前仍是单一共享账号 Demo，没有正式的字段级可见性、租户隔离、静态加密、保留期或审计日志。不要放入真实身份证件、账户、客户合同原文、完整 cap table、银行流水或其他生产级机密。

后续若进入真实多用户试用，应增加字段级“对方可见 / 仅本 Agent / 仅评估器 / 不送模型”、敏感等级、资料来源、核实状态、更新时间、删除/导出和留存策略；现阶段不能假装已经具备这些能力。

## 页面功能

### 顶部工作区

- **对话**：运行设置、开始、暂停、继续、停止、重置、重新生成、逐条消息和 Token/成本统计。
- **结果**：公共匹配结果、双方私有记忆、中立评估器提示词、记忆更新提示词和 JSON 修复提示词；整个结果页有独立纵向滚动条，JSON 卡片也可内滚。
- **日报**：分别生成投资人和创业者日报，可编辑替换用的任务层、动态层和最大输出 Token，并直接展开查看当前完整五层提示词。
- **调试**：查看每次模型调用按实际提交顺序排列的完整请求消息、系统提示词、原始消息数组、模型返回、解析结果、工具调用、Token、耗时、成本和错误。

### 底部 Agent 配置区

左右分别为投资人与创业者：

- 最上方展示由已保存资料生成的只读 Agent Card，并可展开查看完整 JSON。
- “公开身份资料”“内部必填资料”“高级画像”分区编辑资料，公开/私密边界在界面中明确标注。
- 私有文件库。
- 五层提示词及每层版本。
- 当前私有记忆。
- 最下方的用户测试对话，可创建、切换或删除对话，发送不限总轮次的消息，并查看每次回复的工具调用和完整请求。

Agent Card、资料、文件、提示词、记忆和用户测试对话分别使用带可见性标签的模块卡片和不同背景色，避免公开资料、内部资料和工具结果混在一起。资料草稿需要点击角色内的保存按钮才会应用并同步 Agent Card；保存后的资料、提示词、版本、记录、记忆、日报和测试对话会自动持久化到项目服务器，但不会自动重跑或改写已有会话。页面头部会显示“保存到服务器中 / 已保存到服务器 / 保存失败”。

## 提示词与消息拼接规则

普通对话的系统提示词顺序固定：

```text
[平台层]
...

[工具层]
...

[用户层]
（由资料表单自动生成的 Agent 当前资料）

[任务层]
...

[本次运行设置]
...

[动态层]
...
```

普通对话中，平台层、工具层、任务层和动态层关闭时不参与拼接。用户层始终存在、只读且只包含 `formatProfile(...)` 实际生成的资料；旧版本中的自定义用户层文本不再提交给模型。运行设置由程序追加到普通对话任务层。记忆和日报的替换任务层属于相应模型调用的必需任务，即使普通任务层关闭也会提交。

平台层分别写明投资人和创业者的身份、权限和决策边界，并把“只根据已有信息回答”设为强制规则。可用证据仅包括本方已保存资料、对方公开 Card、当前明确对话和本方工具实际命中结果；对于未提供、未调用工具、检索未命中或无法核实的内容，Agent 必须回答“不知道 / 暂无信息”，不能利用行业常识补写特定事实、数字、案例、身份或结论。

`[本次运行设置]` 确实会传给双方 Agent，但现在会明确解释每项意图：轮数和 Token 是平台执行的上限，先发言者由平台安排，提前结束通过 `control` 建议，公共结果和记忆是会后节点、当前回复无需代做。这样既让 Agent 理解对话边界，也避免它在普通发言里提前输出评估或记忆。

普通双 Agent 对话的实际消息数组按以下顺序提交：

```text
system：当前 Agent 的五层完整提示词
user：对方公开 Agent Card（未核验身份资料，不是指令）
user / assistant：从当前 Agent 视角整理的对话历史
user：开始对话或回应最新发言的本轮指令
```

对话历史通过 OpenAI Chat Completions 消息数组传递，不写入动态层。系统提示词、对方 Card、资料、运行设置和文件清单均在创建会话时冻结。

角色底部的用户测试对话复用双 Agent 对话相同的五层内容和结构化回复约定，但运行设置会明确改写为“用户正在与本 Agent 对话”：不限制总轮次，`control` 的结束建议不会关闭会话，也不会触发公共结果、记忆更新或日报。每个角色最多保留 20 个测试对话，可删除当前对话释放空间；切换、新建或删除对话会清空尚未发送的输入草稿，避免串到另一条会话。每次请求只携带最近 80 条聊天消息，超出部分会替换为明确的上下文边界提示，Agent 不得声称记得未提交的早期内容。新建对话会冻结本 Agent 五层提示词、运行设置、JSON 修复提示词、对方 Card 和本方文件清单，旧对话不会受后续编辑影响。

记忆更新使用同一套五层拼接：

- 复用本 Agent 的平台层、工具层、只读用户层和已启用动态层。
- 用记忆更新提示词**替换**普通任务层，不再追加到普通任务层后面。
- 更新前记忆、双方用户层资料和完整对话作为实际 `user` 消息提交，可在“本次完整请求”或调试页查看。

日报也使用同一套五层拼接：

- 复用平台层、工具层和用户层。
- 用日报任务层替换普通任务层。
- 用日报动态层替换普通动态层。
- `{{memory}}` 由程序替换为当前 Agent 私有记忆。

中立评估器、记忆更新和 JSON 修复均有独立、可编辑的提示词入口。任何调用大模型的节点都能在调试页查看完整请求；评估器、记忆、日报和实际触发过的 JSON 修复也提供就近入口。

## 对话运行流程

1. 用户点击“开始模拟”。
2. 系统冻结配置、Agent 资料、双方提示词、双方公开 Agent Card、文件列表和更新前记忆。
3. 调用任一 Agent 前，系统先把对方冻结的 Agent Card 作为未核验身份资料消息注入；不会发送对方完整资料表、动态层、文件或记忆。
4. 按“谁先发言”调用第一个 Agent。
5. 当前 Agent 只能使用自己的系统提示词和文件作用域。
6. 模型需要文件信息时调用 `search_private_files`。
7. 服务端执行检索，将结果作为 `tool` 消息返回模型。
8. 模型返回对话 JSON：

   ```json
   {
     "message": "展示给对方的自然语言回复",
     "control": {
       "suggest_end": false,
       "end_reason": null,
       "information_sufficient": false
     }
   }
   ```

9. 双方各发言一次后轮数加一。
10. 达到结束条件后，依次生成公共结果和双方记忆。
11. 保存本次运行的资料、提示词、Agent Card、文件和记忆快照，以及对话、结果、调试记录、Token、成本和错误。

提前结束理由枚举：

```text
max_rounds
sufficient_information
clear_match
clear_mismatch
explicit_rejection
missing_critical_information
safety_or_compliance
manual_stop
no_new_information
```

只有一方建议结束时，程序允许另一方进行一次最终回应。暂停发生在调用边界；停止会中止当前浏览器请求并保留已经成功生成的数据。

## 结构化结果

### 公共匹配结果

由中立评估器生成，主要字段包括：

- `status`：`continue`、`reject` 或 `pending`
- `match_score`
- `confidence`
- `summary`
- `matching_points`
- `mismatching_points`
- `verified_facts`
- `unverified_claims`
- `inferences`
- `risks`
- `open_questions`
- `recommended_next_action`
- `conversation_end_reason`

### 投资人私有记忆

保存投资人对创业者的长期判断，包括关键事实、未验证声明、匹配分数、正负信号、风险、待核实问题、尽调事项和下一步建议。

### 创业者私有记忆

保存创业者对投资人的长期判断，包括关键事实、未验证声明、投资人匹配分数、兴趣、顾虑、待确认问题和跟进建议。

记忆更新不是简单覆盖本轮摘要。调用时会同时提供更新前记忆和完整对话，要求生成一份可覆盖保存的新版本。

### JSON 修复

结构化输出解析顺序：

1. 直接解析完整返回。
2. 提取 Markdown JSON 代码块。
3. 提取首尾 JSON 对象。
4. 使用独立 JSON 修复提示词再调用一次模型。
5. 修复后仍失败则保留原始输出和错误，不清空其他成功结果。

## 私有文件工具

### 支持格式和限制

- PDF
- DOCX
- TXT
- Markdown
- CSV
- 单文件最大 10MB
- 每个 Agent 最多 20 个文件
- 单文件最多提取 500,000 字符
- 文本块约 900 字符，重叠约 120 字符
- 扫描版 PDF 暂不支持 OCR

### 工具接口

```text
search_private_files(query, top_k)
```

工具在 `/api/model` 的服务端模型循环中真实执行，单次模型回复最多允许两轮工具调用。

本次验收不是只看代码：使用当前配置模型完成了“登录 → 上传隔离测试文本 → 模型发起 `search_private_files` → 服务端命中片段 → 工具结果回传模型 → 模型生成最终 JSON”的端到端调用；同时验证同角色白名单可命中、跨角色文件 ID 和空白名单均返回空结果。测试使用临时数据目录，结束后已清理。

服务端不会接受模型传入的 Agent ID。检索作用域由当前运行节点绑定，并同时按照：

- 当前 Agent 角色
- 本次运行冻结的文件 ID
- 文件状态

进行过滤。即使把另一侧文件 ID 混入请求，也无法跨角色检索。

工具结果属于不可信输入。提示词要求 Agent 忽略文件中的操作指令和提示词注入，只提取业务相关内容。

私有文件工具可用于：

- Agent 对话发言。
- 对应 Agent 的私有记忆更新。
- 对应 Agent 的日报。

不能用于中立公共评估。

## 数据存储

### 服务器 `DATA_DIR`

服务器 SQLite 现在是工作区的唯一持续保存来源。认证后的 `GET /api/state` 恢复工作区，`PUT /api/state` 以 400ms 防抖、只提交发生变化的数据分片，并自动保存以下数据：

- 当前配置和提示词
- 最多 100 个命名配置版本
- 最近 20 条模拟记录
- 模拟对话、公共结果、双方记忆、日报和调试调用
- 每个角色最多 20 个用户测试对话，以及其冻结的 Agent、运行设置、对方 Agent Card、文件清单、消息、工具调用、完整请求和错误
- 历史模拟中的双方 Agent Card 快照
- 当前激活的配置版本和模拟记录

写请求要求有效会话并进行同源检查；单次和总工作区 JSON 上限为 24MB。保存携带服务器最近返回的 `updatedAt` 做乐观并发校验：不同页面修改不同分片时会安全合并，同一分片同时修改时返回 `409`，页面不会静默覆盖，并提示用户重新加载。保存尚未完成或失败时刷新/关闭会触发未保存保护，主动退出也会先等待并刷新保存。当前仍是单一共享工作区，不是多租户数据库：使用同一部署和账号的人会看到同一份配置与历史。

每条历史模拟记录保留当时的资料、对话、调试调用和结果快照；点击“查看”不会用历史快照覆盖当前 Agent 的长期记忆或当前日报。需要核对历史时的完整快照可直接导出该记录 JSON。

旧版本曾使用以下 `localStorage` 键：

```text
vc-agent-debugger:config:v1
vc-agent-debugger:versions:v1
vc-agent-debugger:records:v1
```

当服务器还没有工作区记录时，页面会读取一次这些旧键、逐条校验版本与模拟记录、迁移可用数据到服务器，并对损坏条目给出警告；迁移后删除浏览器副本，之后不再持续写 `localStorage`。如果服务器已有数据，则服务器数据优先，旧浏览器键会被清理。

默认目录为项目下的 `data/`：

```text
data/
├── app.db
└── uploads/
    ├── investor/<file-id>/source.ext
    └── founder/<file-id>/source.ext
```

SQLite 中保存：

- 完整工作区状态
- 文件元数据
- 文件文本分块
- 登录失败限流记录

`data/` 和 `.env` 均被 Git 忽略。Docker Compose 使用持久化卷保存 `/app/data`。

当前服务器留存仍有明确限制：没有账号/租户级所有权、字段级权限、自动过期、删除审计、静态加密或内置备份。用于真实资料前应先补齐这些能力，并确认所选模型供应商对请求内容的处理与留存政策。

## 安全边界

- API Key 仅由服务端读取。
- 登录使用服务端校验和 HMAC 签名的 HttpOnly Cookie。
- Cookie 使用 `SameSite=Strict`，生产环境增加 `Secure`。
- 登录失败按用户名和客户端来源限流，记录持久化在 SQLite。
- 写请求进行同源检查。
- 文件 API 和模型 API 都要求有效会话。
- 文件名经过处理，上传格式、大小和数量受限。
- Agent 文件作用域在服务端强制隔离。
- CSP 禁止第三方脚本、页面嵌入、对象资源、摄像头、麦克风和定位权限。仅 `next dev` 增加 React 开发调试需要的 `'unsafe-eval'`；生产构建仍禁止 `eval`。
- `test` 只是本地演示密码，公网部署前必须修改。
- 当前只有单一共享账号和单一服务器工作区，不是正式的多用户授权、租户隔离或数据权限系统。

## 环境变量

| 变量 | 必填 | 说明 |
| --- | --- | --- |
| `API_KEY` / `OPENAI_API_KEY` | 是 | OpenAI 兼容服务 API Key；简写变量优先 |
