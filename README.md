# 创投社区数字分身对话调试器

这是一个桌面端 Web Demo，用来验证“可配置数字分身能否完成有价值的投融资初筛，并产出可解释、可保存的结构化结果”。

它不是完整的创投社区，也不是自动投资系统。项目当前应优先保证提示词流、资料流、工具流、对话流、结果流和调试流清晰可靠，避免为了产品完整度引入无关功能。

---

## 给后续 Agent 的项目约束

后续修改本项目时，请先遵守以下约束：

1. **不要把项目扩展成完整创投社区。** 当前核心是双 Agent 对话调试与结果验证。
2. **不要代表真实用户作投资、融资、签约、接受估值或资金承诺。** 所有输出只用于初筛和人工决策辅助。
3. **投资人和创业者必须隔离。** 当前 Agent 不能获得对方的隐藏提示词、私有文件或私有记忆。
4. **所有模型调用必须经过服务端。** 禁止把 API Key、Base URL 或模型请求逻辑暴露到浏览器。
5. **运行开始后必须使用快照。** 编辑 Agent、提示词、文件或记忆不能改写正在运行或已经完成的对话。
6. **历史对话是独立消息历史。** 不要把完整历史塞入动态层。
7. **结构化结果解析失败时不能静默丢失。** 必须保留原始输出、修复调用和错误信息。
8. **保持向后兼容。** 修改 `AppConfig`、模拟记录或提示词结构时，同步更新 `migrateConfig` 和旧本地数据恢复逻辑。
9. **不要自动部署。** 除非用户明确要求，否则只在本地修改、构建和测试。
10. **不要提交 `.env` 或 `data/`。** 它们包含凭据、上传文件、索引或运行数据。

## Demo 要验证什么

### 核心验证 1：分层提示词能否稳定控制两个 Agent

投资人与创业者各自拥有独立的五层提示词：

1. 平台层
2. 工具层
3. 用户层
4. 任务层
5. 动态层

每层可以启用、关闭、编辑、恢复默认，并可独立保存多个命名版本。系统按照固定顺序生成最终提示词，Agent 基本资料由程序自动追加到用户层之后。

需要验证：

- 不同角色、资料和提示词是否会产生可预期的行为差异。
- 平台规则是否能抵抗低优先级内容和文件中的提示词注入。
- 工具层关闭后，模型是否停止使用文件工具。
- 编辑是否只影响下一次模拟。
- 页面显示的“本次实际提示词”是否与真正提交给模型的内容一致。

### 核心验证 2：两个 Agent 能否完成有价值的投融资初筛

投资人 Agent 应判断项目与投资偏好的匹配程度，主动了解团队、市场、产品、业务数据、融资需求和资金用途。

创业者 Agent 应表达项目价值、融资诉求和资源需求，同时判断投资人的阶段、赛道、资源能力和合作方式是否适合。

需要验证：

- 双方是否优先询问影响匹配判断的问题。
- 是否避免重复提问、无效寒暄和一次提出过多问题。
- 是否能区分已知事实、对方声明、文件内容、推断和未知信息。
- 信息不足时是否承认不足，而不是虚构数据。
- 是否能通过结构化控制字段稳定提前结束。
- 一来一回是否正确计为一轮。

### 核心验证 3：对话结束后能否生成可入库结论和记忆

对话结束后有三个主要后处理节点：

- 中立评估器生成公共匹配结果。
- 投资人 Agent 更新关于创业者的长期私有记忆。
- 创业者 Agent 更新关于投资人的长期私有记忆。

需要验证：

- 公共结果是否中立、可解释且严格区分事实、声明和推断。
- 私有记忆是否保留仍有效的旧信息、合并重复内容，并加入本轮长期有用的新信息。
- 新旧记忆冲突时是否保留冲突并标记待核实。
- JSON 是否符合约定结构，失败时是否可修复和追踪。
- 结果是否能编辑、复制、下载并随模拟记录保存。

### 扩展验证：私有文件工具和日报

当前还验证两个扩展概念：

- Agent 能否通过真实工具从自己的服务器文件库中检索资料，同时保持双方隔离。
- Agent 能否基于当前私有记忆生成可调试的工作日报。

这两项服务于核心验证，不应演变成通用知识库或办公自动化平台。

主要资料包括公司、项目介绍、赛道、融资轮次、融资金额、产品阶段、团队、业务数据、商业模式、客户、估值、优势、资金用途、资源需求、投资方排除和当前挑战。

模型运行时可以看到：

### 中立评估器

中立评估器不是第三个对话 Agent，只在对话结束后运行。它只能看到双方用于评估的公开资料、完整对话、Agent ID 和结束原因。

中立评估器不能调用私有文件工具，也不能读取双方私有记忆。

## 页面功能

### 顶部工作区

- **对话**：运行设置、开始、暂停、继续、停止、重置、重新生成、逐条消息和 Token/成本统计。
- **结果**：公共匹配结果、双方私有记忆、中立评估器提示词、记忆更新提示词和 JSON 修复提示词。
- **日报**：分别生成投资人和创业者日报，可编辑任务层、动态层和最大输出 Token。
- **调试**：查看每次模型调用的完整提示词、历史、原始输出、解析结果、工具调用、Token、耗时、成本和错误。

### 底部 Agent 配置区

左右分别为投资人与创业者：

- 注册基础必填资料。
- 高级画像选填资料。
- 私有文件库。
- 五层提示词及每层版本。
- 当前私有记忆。

资料编辑、提示词和配置会自动保存到当前浏览器，但不会自动重跑已有对话。

## 提示词与消息拼接规则

普通对话的系统提示词顺序固定：

```text
[平台层]
...

[工具层]
...

[用户层]
...

[Agent 当前资料]
...

[任务层]
...

[本次运行设置]
...

[动态层]
...
```

关闭的层不参与拼接。Agent 资料只在用户层启用时追加。运行设置由程序追加到任务层。

对话历史通过 OpenAI Chat Completions 消息数组传递，不写入动态层。

日报使用特殊拼接：

- 复用平台层、工具层和用户层。
- 用日报任务层替换普通任务层。
- 用日报动态层替换普通动态层。
- `{{memory}}` 由程序替换为当前 Agent 私有记忆。

中立评估器、记忆更新和 JSON 修复均有独立、可编辑的提示词入口。

## 对话运行流程

1. 用户点击“开始模拟”。
2. 系统冻结配置、Agent 资料、双方提示词、文件列表和更新前记忆。
3. 按“谁先发言”调用第一个 Agent。
4. 当前 Agent 只能使用自己的系统提示词和文件作用域。
5. 模型需要文件信息时调用 `search_private_files`。
6. 服务端执行检索，将结果作为 `tool` 消息返回模型。
7. 模型返回对话 JSON：

   ```json
   {
     "message": "展示给对方的自然语言回复",
     "control": {
       "suggest_end": false,
       "end_reason": null,
       "information_sufficient": false
     }
   }
   ```

8. 双方各发言一次后轮数加一。
9. 达到结束条件后，依次生成公共结果和双方记忆。
10. 保存本次运行的快照、对话、结果、调试记录、Token、成本和错误。

提前结束理由枚举：

```text
max_rounds
sufficient_information
clear_match
clear_mismatch
explicit_rejection
missing_critical_information
safety_or_compliance
manual_stop
no_new_information
```

只有一方建议结束时，程序允许另一方进行一次最终回应。暂停发生在调用边界；停止会中止当前浏览器请求并保留已经成功生成的数据。

## 结构化结果

### 公共匹配结果

由中立评估器生成，主要字段包括：

- `status`：`continue`、`reject` 或 `pending`
- `match_score`
- `confidence`
- `summary`
- `matching_points`
- `mismatching_points`
- `verified_facts`
- `unverified_claims`
- `inferences`
- `risks`
- `open_questions`
- `recommended_next_action`
- `conversation_end_reason`

### 投资人私有记忆

保存投资人对创业者的长期判断，包括关键事实、未验证声明、匹配分数、正负信号、风险、待核实问题、尽调事项和下一步建议。

### 创业者私有记忆

保存创业者对投资人的长期判断，包括关键事实、未验证声明、投资人匹配分数、兴趣、顾虑、待确认问题和跟进建议。

记忆更新不是简单覆盖本轮摘要。调用时会同时提供更新前记忆和完整对话，要求生成一份可覆盖保存的新版本。

### JSON 修复

结构化输出解析顺序：

1. 直接解析完整返回。
2. 提取 Markdown JSON 代码块。
3. 提取首尾 JSON 对象。
4. 使用独立 JSON 修复提示词再调用一次模型。
5. 修复后仍失败则保留原始输出和错误，不清空其他成功结果。

## 私有文件工具

### 支持格式和限制

- PDF
- DOCX
- TXT
- Markdown
- CSV
- 单文件最大 10MB
- 每个 Agent 最多 20 个文件
- 单文件最多提取 500,000 字符
- 文本块约 900 字符，重叠约 120 字符
- 扫描版 PDF 暂不支持 OCR

### 工具接口

```text
search_private_files(query, top_k)
```

工具在 `/api/model` 的服务端模型循环中真实执行，单次模型回复最多允许两轮工具调用。

服务端不会接受模型传入的 Agent ID。检索作用域由当前运行节点绑定，并同时按照：

- 当前 Agent 角色
- 本次运行冻结的文件 ID
- 文件状态

进行过滤。即使把另一侧文件 ID 混入请求，也无法跨角色检索。

工具结果属于不可信输入。提示词要求 Agent 忽略文件中的操作指令和提示词注入，只提取业务相关内容。

私有文件工具可用于：

- Agent 对话发言。
- 对应 Agent 的私有记忆更新。
- 对应 Agent 的日报。

不能用于中立公共评估。

## 调试与可观测性

调用类型包括：

```text
investor_turn
founder_turn
public_evaluation
investor_memory
founder_memory
investor_daily_report
founder_daily_report
json_repair
```

每次调用记录：

- 调用类型和角色
- 当前轮次
- 完整系统提示词
- 层级开关
- Agent 资料快照
- 模型消息历史
- 原始模型返回
- 解析结果
- 工具查询和命中片段
- 开始与结束时间
- 耗时
- 输入、输出和总 Token
- Token 是否为估算值
- 估算成本
- 成功状态和错误信息

如果兼容服务返回 `usage`，优先使用真实 Token；否则采用粗略估算。输入和输出单价由页面设置，默认均为 0。

## 数据存储

### 浏览器 `localStorage`

以下数据按浏览器、域名和设备保存：

- 当前配置
- 提示词版本
- 最近 20 条模拟记录
- 对话与三类结果
- 日报
- 调试记录
- 当前长期记忆状态（通过加载记录恢复）

主要键：

```text
vc-agent-debugger:config:v1
vc-agent-debugger:versions:v1
vc-agent-debugger:records:v1
```

注意：当前模拟记录和记忆不是服务器多用户数据库。不同合伙人的浏览器不会自动共享这些数据，除非导出后导入或未来增加服务端记录存储。

### 服务器 `DATA_DIR`

默认目录为项目下的 `data/`：

```text
data/
├── app.db
└── uploads/
    ├── investor/<file-id>/source.ext
    └── founder/<file-id>/source.ext
```

SQLite 中保存：

- 文件元数据
- 文件文本分块
- 登录失败限流记录

`data/` 和 `.env` 均被 Git 忽略。Docker Compose 使用持久化卷保存 `/app/data`。

## 安全边界

- API Key 仅由服务端读取。
- 登录使用服务端校验和 HMAC 签名的 HttpOnly Cookie。
- Cookie 使用 `SameSite=Strict`，生产环境增加 `Secure`。
- 登录失败按用户名和客户端来源限流，记录持久化在 SQLite。
- 写请求进行同源检查。
- 文件 API 和模型 API 都要求有效会话。
- 文件名经过处理，上传格式、大小和数量受限。
- Agent 文件作用域在服务端强制隔离。
- 默认 CSP 禁止第三方脚本、页面嵌入、对象资源、摄像头、麦克风和定位权限。
- `test` 只是本地演示密码，公网部署前必须修改。
- 当前只有单一共享账号，不是正式的多用户授权或数据权限系统。

## 环境变量

| 变量 | 必填 | 说明 |
| --- | --- | --- |
| `API_KEY` / `OPENAI_API_KEY` | 是 | OpenAI 兼容服务 API Key；简写变量优先 |
| `BASE_URL` / `OPENAI_BASE_URL` | 是 | API Base URL，例如 `https://api.openai.com/v1` |
| `MODEL` / `OPENAI_MODEL` | 是 | 对话、评估、记忆、日报和修复共用模型 |
| `OPENAI_TIMEOUT_MS` | 否 | 单次服务端模型请求超时，默认 90000ms |
| `DATA_DIR` | 否 | 文件、SQLite 和限流数据目录，默认 `./data` |
| `AUTH_USERNAME` | 否 | 登录用户名，默认 `user` |
| `AUTH_PASSWORD` | 否 | 登录密码，默认 `test`；公网必须修改 |
| `AUTH_SESSION_SECRET` | 生产必填 | HMAC 会话密钥，至少 32 个高熵字符 |
| `APP_ORIGIN` | 公网建议 | 唯一外部站点地址，用于严格同源校验 |
| `TRUST_PROXY` | 否 | 仅在可信反向代理清洗转发头时设为 `1` |

不同 OpenAI 兼容服务价格不同，因此模型单价不通过环境变量写死，而在页面本地配置。

## Docker 部署

部署前必须：

1. 修改 `AUTH_PASSWORD`。
2. 设置至少 32 字符的随机 `AUTH_SESSION_SECRET`。
3. 设置正确的 `APP_ORIGIN`。
4. 配置模型 API 参数。

启动：

```bash
docker compose up -d --build
```

Compose 默认只将端口绑定到 `127.0.0.1:3000`，应在同机使用 Nginx 或 Caddy 提供 HTTPS 反向代理。

持久化卷：

```text
venture-agent-data → /app/data
```

上线试用前还应考虑：

- IP 白名单或 VPN
- 反向代理全局限流
- 请求体大小限制
- HSTS
- 数据卷备份
- 定期轮换密码和会话密钥
