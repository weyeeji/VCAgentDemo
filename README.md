# 创投社区数字分身对话调试器

这是一个桌面端 Web Demo，用来验证“可配置数字分身能否完成有价值的投融资初筛，并产出可解释、可保存的结构化结果”。

Agent 独立 Memory、任务状态、用户指令确认和运行快照的设计与实现见 [Agent Memory 系统实现](./docs/Agent-Memory系统实现.md)。

它不是完整的创投社区，也不是自动投资系统。项目当前应优先保证提示词流、资料流、工具流、对话流、结果流和调试流清晰可靠，避免为了产品完整度引入无关功能。

---

## 给后续 Agent 的项目约束

后续修改本项目时，请先遵守以下约束：

1. **不要把项目扩展成完整创投社区。** 当前核心是双 Agent 对话调试与结果验证。
2. **不要代表真实用户作投资、融资、签约、接受估值或资金承诺。** 所有输出只用于初筛和人工决策辅助。
3. **投资人和创业者必须隔离。** 当前 Agent 不能获得对方的隐藏提示词、私有文件或私有记忆。
4. **所有模型调用必须经过服务端。** 禁止把 API Key、Base URL 或模型请求逻辑暴露到浏览器。
5. **运行开始后必须使用快照。** 编辑 Agent、提示词、文件或记忆不能改写正在运行或已经完成的对话。
6. **历史对话是独立消息历史。** 不要把完整历史塞入动态层。
7. **结构化结果解析失败时不能静默丢失。** 必须保留原始输出、修复调用和错误信息。
8. **保持向后兼容。** 修改 `AppConfig`、模拟记录或提示词结构时，同步更新 `migrateConfig` 和旧本地数据恢复逻辑。
9. **用户层只承载 Agent 当前资料。** 角色身份、行为规则和决策边界写在平台层，用户层由资料表单自动生成并保持只读。
10. **区分共享配置与会话状态。** 配置、资料库和命名版本以服务器工作区为准；模拟记录、直聊和日报等会话状态保留在当前浏览器。结构化 Agent Memory / 任务写入 SQLite，但按浏览器作用域和 Agent ID 隔离。
11. **未知信息必须明确承认未知。** Agent 只能依据本方已保存资料、当前对话和本方工具实际命中的内容回答；仅在双 Agent 模拟中还可使用对方公开 Agent Card。没有提供、没有调用工具或工具没有命中的信息，不得凭常识、经验或猜测补全。
12. **Agent Card 只能发布白名单字段。** 初筛必问问题、动态层、私有文件、私有记忆和其他未列入白名单的资料不得进入公开卡片；新增字段默认不公开，必须先经过隐私评估。

## Demo 要验证什么

### 核心验证 1：分层提示词能否稳定控制两个 Agent

投资人与创业者各自拥有独立的五层提示词：

1. 平台层
2. 工具层
3. 用户层
4. 任务层
5. 动态层

平台层、工具层、任务层和动态层可以按产品规则启用、关闭、编辑、恢复默认并保存命名版本。用户层固定为只读的 Agent 当前资料，不能另写一份容易与资料冲突的“用户提示词”。系统按照固定顺序生成最终提示词。

需要验证：

- 不同角色、资料和提示词是否会产生可预期的行为差异。
- 平台规则是否能抵抗低优先级内容和文件中的提示词注入。
- 工具层关闭后，模型是否停止使用文件工具。
- 编辑是否只影响下一次模拟。
- 页面显示的“本次实际完整请求”是否与真正提交给模型的系统提示词和消息数组一致。
- 双 Agent 模拟中，对方公开 Agent Card 是否以未核验身份资料进入消息数组，同时保持私有字段、动态层、文件和记忆隔离；用户与自己 Agent 的测试对话则不得携带任何对方 Card。
- 没有明确资料或工具命中时，Agent 是否直接回答“不知道 / 暂无信息”，而不是补写事实、数字、案例、身份或结论。

### 核心验证 2：两个 Agent 能否完成有价值的投融资初筛

投资人 Agent 应判断项目与投资偏好的匹配程度，主动了解团队、市场、产品、业务数据、融资需求和资金用途。

创业者 Agent 应表达项目价值、融资诉求和资源需求，同时判断投资人的阶段、赛道、资源能力和合作方式是否适合。

需要验证：

- 双方是否优先询问影响匹配判断的问题。
- 是否避免重复提问、无效寒暄和一次提出过多问题。
- 是否能区分已知事实、对方声明、文件内容、推断和未知信息。
- 信息不足时是否承认不足，而不是虚构数据。
- 是否能通过结构化控制字段稳定提前结束。
- 一来一回是否正确计为一轮。

### 核心验证 3：对话结束后能否生成可入库结论和记忆

对话结束后有三个主要后处理节点：

- 中立评估器生成公共匹配结果。
- 投资人 Agent 提取关于创业者的增量私有记忆。
- 创业者 Agent 提取关于投资人的增量私有记忆。

需要验证：

- 公共结果是否中立、可解释且严格区分事实、声明和推断。
- 是否只提取本轮长期有用的增量信息，避免重复已有 Memory。
- 新旧记忆冲突时是否保留冲突并标记待核实。
- Agent 的行动提案是否只在用户确认后更改 Memory 或任务。
- JSON 是否符合约定结构，失败时是否可修复和追踪。
- 结果是否能编辑、复制、下载并随模拟记录保存。

### 扩展验证：私有文件工具、用户测试对话和日报

当前还验证三个扩展概念：

- Agent 能否通过真实工具从自己的服务器文件库中检索资料，同时保持双方隔离。
- 用户能否分别与投资人或创业者 Agent 进行不限总轮次的测试对话，并检查真实工具调用和完整模型请求。
- Agent 能否基于最新结构化 Memory 和任务生成可调试的工作日报。

这三项服务于核心验证，不应演变成通用知识库或办公自动化平台。

## 产品角色与可见信息

### 投资人 Agent

主要资料包括个人与机构信息、关注赛道、投资阶段、机构与投资规模、目前投资状态、近期活动目标、具体投资方向、判断信号、投资禁忌和 Agent 报告形式；公开层级以 DOCX 的三类模块为准。

模型运行时可以看到：

- 投资人自己的最终组合提示词。
- 投资人自己的资料快照。
- 双 Agent 模拟中，对方在会话创建时冻结的公开 Agent Card；其中内容只按对方自报、未经核验的声明处理。
- 本次运行设置。
- 已产生的双方对话历史。
- 投资人自己的文件检索结果。
- 本次运行开始时冻结的投资人 Working Context（已确认决策、当前任务和相关 Memory）。

不能看到创业者的隐藏提示词、私有文件或创业者私有记忆。

### 创业者 Agent

主要资料包括姓名与公司、项目介绍与亮点、核心产品、赛道、融资轮次、近期活动目标、融资计划、前期融资、公司介绍、重点询问的机构信息、人才/业务需求和 Agent 报告形式；公开层级以 DOCX 的三类模块为准。

模型运行时可以看到：

- 创业者自己的最终组合提示词。
- 创业者自己的资料快照。
- 双 Agent 模拟中，对方在会话创建时冻结的公开 Agent Card；其中内容只按对方自报、未经核验的声明处理。
- 本次运行设置。
- 已产生的双方对话历史。
- 创业者自己的文件检索结果。
- 本次运行开始时冻结的创业者 Working Context（已确认决策、当前任务和相关 Memory）。

不能看到投资人的隐藏提示词、私有文件或投资人私有记忆。

双 Agent 普通对话节点不会拿到对方的完整资料表，只会在对话开始前收到对方公开 Agent Card，再从后续对话中了解其他信息；会后记忆提取节点会收到双方用户层资料、本方冻结 Working Context 和完整对话，但仍不能读取对方隐藏提示词、私有文件或私有记忆。角色底部的用户测试对话模拟创建者/管理者用户与自己的 Agent 交流，使用聊天区可编辑的专用任务层完全替换普通任务层，每轮都读取本 Agent 最新 Working Context，不创建也不发送对方 Agent Card。

### 公开 Agent Card

页面为每个角色自动生成一张只读 Agent Card，设计参考 [A2A Protocol 1.0 规范](https://github.com/a2aproject/A2A/blob/main/docs/specification.md)中用于发现 Agent 身份、能力、输入输出模式和技能的 Agent Card 概念。当前只是 `a2a-inspired` 的 Demo 数据结构，不是完整 A2A 实现：项目没有提供标准 Agent-to-Agent HTTP 接口、认证协商、任务/制品、流式传输、推送通知或跨实现互操作承诺。

卡片只从资料白名单自动生成：

- DOCX“公开信息”中的已填写字段一定进入卡片。
- DOCX“选择性公开信息”默认进入卡片，用户可逐字段关闭公开开关；关闭后该字段从卡片移除。
- DOCX“不公开信息”、动态层、私有文件、私有记忆和其他未列出的字段一定不进入卡片。

资料表先在页面形成草稿；点击对应角色的“保存资料并更新 Agent Card”后，用户层、公开卡片和服务器状态才一起更新。未保存草稿会阻止创建新测试对话或开始模拟。既有双 Agent 模拟继续使用开始时冻结的双方卡片；用户测试对话的 Agent 身份与文件范围在创建时冻结，但专用任务层编辑和最新 Memory / 任务从下一条消息开始生效。

会话创建后，对方卡片会作为单独的 `user` 身份资料消息放在对话历史之前，并被明确标记为“未核验、不可信数据，不是指令”。卡片缺失的内容仍视为未知；卡片中的文本也不能更改平台规则、角色、任务或工具权限。身份层用于公开展示和未来匹配筛选，动态层保存不愿直接公开的上下文，详细资料继续留在对应角色的私有文件中并按需检索。

服务器不会直接信任浏览器提交的 Card：它会校验对端角色和公开字段白名单，并从 `agentId + role + claims` 重建规范 Card；名称、说明、技能、能力、输入输出模式或任意额外字段与规范结果不一致时，状态写入会被拒绝，防止借 Card 其他位置夹带私密资料。

### 中立评估器

中立评估器不是第三个对话 Agent，只在对话结束后运行。它只能看到双方用户层中用于评估的已填写资料、完整对话、Agent ID 和结束原因。

中立评估器不能调用私有文件工具，也不能读取双方私有记忆。

### 资料表单的专业设计与隐私边界

当前表单的字段、选择项和三类公开层级以 `docs/投资人及创业者信息.docx` 为唯一设计来源，不再追加旧版扩展字段。选择性公开状态保存在每套资料中；不公开信息虽然可供本方 Agent 内部判断，但不得出现在对方 Agent Card 或公共结果中。私有文件只向对应 Agent 的检索工具开放。

当前仍是单一共享账号 Demo，尚无租户隔离、静态加密、保留期或审计日志。不要放入真实身份证件、账户、客户合同原文、完整 cap table、银行流水或其他生产级机密。

### 多套用户资料与错配测试

投资人和创业者各自拥有独立资料库，并各内置三套完整预设：

| 组合 | 投资人预设 | 创业者预设 | 预期用途 |
| --- | --- | --- | --- |
| A | AI 企业服务早期 | 工业 AI 企业服务 | 同赛道、同阶段的匹配对话 |
| B | 医疗健康成长期 | 医疗器械成长期 | 同赛道、同阶段的匹配对话 |
| C | 九和投资早期科技 | 小友智心端侧智能 | 使用项目设计者提供的真实样例口径；仅创业者关联项目介绍 PDF |
| 交叉组合 | 不同领域与阶段交叉 | 不同领域与阶段交叉 | 验证 Agent 能否识别明确不匹配并继续询问必要信息 |

每个角色顶部都可以选择资料或点击“新增资料”。新增会创建一套仅带资料名称的空白 DOCX 字段表。同角色资料名称不能重复，每侧最多保留 20 套。公开 Agent Card 由当前资料和逐字段公开开关生成且不可直接编辑。修改资料字段后必须点击保存，Card 和只读用户层才会同步更新。

资料切换不是只换一组表单文字。每套资料都有独立 Agent ID，并独立保存动态层、私有文件关联、结构化 Memory / 任务、日报和用户测试对话；平台层、工具层、普通任务层等产品级配置继续由同一角色共享。Memory 可以通过 `counterparty_id` 按对手方分桶；双 Agent 模拟只注入全局项和当前对手方项，避免跨主体污染。

## 页面功能

### 顶部工作区

- **对话**：运行设置、开始、暂停、继续、停止、重置、重新生成、逐条消息和 Token/成本统计。
- **结果**：公共匹配结果、双方本轮增量记忆、中立评估器提示词、记忆提取提示词和 JSON 修复提示词；整个结果页有独立纵向滚动条，JSON 卡片也可内滚。
- **日报**：分别生成投资人和创业者日报，可编辑替换用的任务层、动态层和最大输出 Token，并直接展开查看当前完整五层提示词。
- **调试**：查看每次模型调用按实际提交顺序排列的完整请求消息、系统提示词、原始消息数组、模型返回、解析结果、工具调用、Token、耗时、成本和错误。

### 底部 Agent 配置区

左右分别为投资人与创业者：

- 最顶部选择当前用户资料；每侧内置 A/B/C 三套预设，也可新增自定义资料。
- 资料选择器下方展示由已保存资料生成的只读 Agent Card，并可展开查看完整 JSON。
- 按“公开信息”“选择性公开信息”“不公开信息”三个固定模块编辑资料。
- 私有文件库。
- 五层提示词及每层版本。
- 结构化 Memory 和任务面板，可搜索、增删改查、确认、归档与恢复。
- 最下方的用户测试对话带有可编辑的专用任务层，可创建、切换或删除对话，发送不限总轮次的消息，查看工具调用和完整请求，并审批 Agent 提出的 Memory / 任务变更。

资料选择器、Agent Card、资料、文件、提示词、Memory / 任务和用户测试对话分别使用带可见性标签的模块卡片和不同背景色，避免公开资料、内部资料和工具结果混在一起。资料草稿需要点击角色内的保存按钮才会应用并同步 Agent Card。配置、资料库和版本保存到共享服务器工作区；Memory / 任务保存到 SQLite 并按浏览器作用域隔离；模拟记录、日报和测试对话是当前浏览器的会话状态。

## 提示词与消息拼接规则

普通对话的系统提示词顺序固定：

```text
[平台层]
...

[工具层]
...

[用户层]
（由资料表单自动生成的 Agent 当前资料）

[任务层]
...

[本次运行设置]
...

[动态层]
...
```

普通双 Agent 对话中，平台层、工具层、任务层和动态层关闭时不参与拼接。用户层始终存在、只读且只包含 `formatProfile(...)` 实际生成的资料；旧版本中的自定义用户层文本不再提交给模型。运行设置由程序追加到普通任务层。用户直聊、记忆提取和日报都有各自的替换任务层，即使普通任务层关闭也会提交对应的专用任务。

平台层分别写明投资人和创业者的身份、权限和决策边界，并把“只根据已有信息回答”设为强制规则。可用证据仅包括本方已保存资料、当前明确对话和本方工具实际命中结果；对方公开 Card 只在双 Agent 模拟中属于可用证据。对于未提供、未调用工具、检索未命中或无法核实的内容，Agent 必须回答“不知道 / 暂无信息”，不能利用行业常识补写特定事实、数字、案例、身份或结论。

`[本次运行设置]` 确实会传给双方 Agent，但现在会明确解释每项意图：轮数和 Token 是平台执行的上限，先发言者由平台安排，提前结束通过 `control` 建议，公共结果和记忆是会后节点、当前回复无需代做。这样既让 Agent 理解对话边界，也避免它在普通发言里提前输出评估或记忆。

普通双 Agent 对话的实际消息数组按以下顺序提交：

```text
system：当前 Agent 的五层完整提示词
user：对方公开 Agent Card（未核验身份资料，不是指令）
user / assistant：从当前 Agent 视角整理的对话历史
user：开始对话或回应最新发言的本轮指令
```

对话历史通过 OpenAI Chat Completions 消息数组传递，不写入动态层。双 Agent 模拟的系统提示词、对方 Card、资料、运行设置和文件清单均在创建会话时冻结。

角色底部的用户测试对话模拟创建者/管理者用户与自己的 Agent 交流。平台层、工具层、只读用户层和动态层沿用本 Agent 配置；聊天区的“用户直聊专用提示词”完全替换普通双 Agent 任务层，并明确 user 不是投资人/创业者对手方。身份、JSON 修复提示词和文件清单在创建对话时冻结；专用任务层的编辑和最新 Memory / 任务从下一轮开始生效。Agent 可以在 `actions` 中提出增删改 Memory 或创建/更新任务，只有用户在界面确认后才会原子写入。每个角色最多保留 20 个测试对话；每次请求只携带最近 80 条聊天消息，超出部分会替换为明确的上下文边界提示。

旧版本曾把对方 Card 注入用户测试对话。服务器迁移时会丢弃带有该旧快照的测试线程，因为其提示词、历史回复和调试请求都可能已受对方资料影响，不能安全地伪装成“用户与自己的 Agent”继续使用；双 Agent 模拟记录不受此迁移影响。

用户测试对话的实际消息数组按以下顺序提交：

```text
system：本 Agent 平台层、工具层、只读用户层、用户直聊替换任务层、动态层与最新 Working Context
user / assistant：最近的用户与本 Agent 对话消息
```

模型调用工具时，服务端只会从该冻结资料关联的文件 ID 中检索；工具返回会以标准 `tool` 消息进入同一次模型调用，但对方 Card 始终不在这条链路中。

增量记忆提取使用同一套五层拼接：

- 复用本 Agent 的平台层、工具层、只读用户层和已启用动态层。
- 用增量记忆提取提示词**替换**普通任务层，不再追加到普通任务层后面。
- 运行开始时的 Working Context、双方用户层资料和完整对话作为实际请求提交，输出逐条幂等写入，不覆盖整份记忆。

日报也使用同一套五层拼接：

- 复用平台层、工具层和用户层。
- 用日报任务层替换普通任务层。
- 用日报动态层替换普通动态层。
- `{{memory}}` 由程序替换为当前 Agent 的结构化 Memory 列表，最新任务另外通过 Working Context 注入任务层。

中立评估器、增量记忆提取和 JSON 修复均有独立、可编辑的提示词入口。任何调用大模型的节点都能在调试页查看完整请求。

日报每次生成都会建立一条独立运行记录并冻结当时的资料、提示词、文件和记忆，不再把新日报或新调试调用追加进已经完成的旧模拟记录。

## 对话运行流程

1. 用户点击“开始模拟”。
2. 系统读取最新结构化 Memory / 任务，并冻结配置、Agent 资料、双方提示词、双方公开 Agent Card、文件列表和双方 Working Context。
3. 调用任一 Agent 前，系统先把对方冻结的 Agent Card 作为未核验身份资料消息注入；不会发送对方完整资料表、动态层、文件或记忆。
4. 按“谁先发言”调用第一个 Agent。
5. 当前 Agent 只能使用自己的系统提示词和文件作用域。
6. 模型需要文件信息时调用 `search_private_files`。
7. 服务端执行检索，将结果作为 `tool` 消息返回模型。
8. 模型返回对话 JSON：

   ```json
   {
     "message": "展示给对方的自然语言回复",
     "control": {
       "suggest_end": false,
       "end_reason": null,
       "information_sufficient": false
     }
   }
   ```

9. 双方各发言一次后轮数加一。
10. 达到结束条件后，依次生成公共结果和双方增量记忆。
11. 保存本次运行的资料、提示词、Agent Card、文件和记忆快照，以及对话、结果、调试记录、Token、成本和错误。

提前结束理由枚举：

```text
max_rounds
sufficient_information
clear_match
clear_mismatch
explicit_rejection
missing_critical_information
safety_or_compliance
manual_stop
no_new_information
```

只有一方建议结束时，程序允许另一方进行一次最终回应。暂停发生在调用边界；停止会中止当前浏览器请求并保留已经成功生成的数据。

## 结构化结果

### 公共匹配结果

由中立评估器生成，主要字段包括：

- `status`：`continue`、`reject` 或 `pending`
- `match_score`
- `confidence`
- `summary`
- `matching_points`
- `mismatching_points`
- `verified_facts`
- `unverified_claims`
- `inferences`
- `risks`
- `open_questions`
- `recommended_next_action`
- `conversation_end_reason`

### 投资人增量私有记忆

提取投资人本轮关于创业者的长期有用信息，并逐条写入 Agent 的结构化 Memory。

### 创业者增量私有记忆

提取创业者本轮关于投资人的长期有用信息，并逐条写入 Agent 的结构化 Memory。

自动提取只能生成 `fact / preference / constraint / note`，程序强制标记为未核实（或保留冲突状态）。`decision` 只能由用户手动写入，或在直聊中确认 Agent 的行动提案后写入。

### JSON 修复

结构化输出解析顺序：

1. 直接解析完整返回。
2. 提取 Markdown JSON 代码块。
3. 提取首尾 JSON 对象。
4. 使用独立 JSON 修复提示词再调用一次模型。
5. 修复后仍失败则保留原始输出和错误，不清空其他成功结果。

## 私有文件工具

### 支持格式和限制

- PDF
- DOCX
- TXT
- Markdown
- CSV
- 单文件最大 10MB
- 每个 Agent 最多 20 个文件
- 单文件最多提取 500,000 字符
- 文本块约 900 字符，重叠约 120 字符
- 扫描版 PDF 暂不支持 OCR

### 工具接口

```text
search_private_files(query, top_k)
```

工具在 `/api/model` 的服务端模型循环中真实执行，单次模型回复最多允许两轮工具调用。

本次验收不是只看代码：使用当前配置模型完成了“登录 → 创建用户与自己 Agent 的对话 → 模型发起 `search_private_files` → 服务端命中 PDF 片段 → 工具结果回传模型 → 模型生成最终 JSON”的端到端调用。投资人 A 资料检索 `INV-AXIS-042` 时真实调用工具 1 次并命中 1 个片段；切到同角色医疗投资人 B 后，同一查询真实调用工具 1 次但命中 0 个片段并明确回答未知。两次完整请求均不含对方 Agent Card。另有服务端校验覆盖跨角色、跨资料现存文件 ID 和空白名单。测试用的正常 PDF 继续保留在当前项目服务器。

服务端不会接受模型传入的 Agent ID。检索作用域由当前运行节点绑定，并同时按照：

- 当前 Agent 角色
- 当前用户资料 ID 及服务器保存的文件关联白名单
- 本次运行冻结的文件 ID
- 文件状态

进行过滤。即使浏览器把另一角色或同角色另一套资料的现存文件 ID 混入请求，服务端也会拒绝该请求，无法跨角色或跨资料检索；历史快照里已经从服务器删除的旧文件 ID 会被安全忽略，不会让整条历史对话失效。

旧版本只有角色级文件库，没有资料 ID。页面首次迁移这类尚未关联的文件时，会固定归入对应角色的 A 预设，而不是归入刷新页面时碰巧选中的资料；之后上传只允许在当前工作区和资料已保存时进行，并自动关联当前资料，日常运行不会把孤儿文件重新认领到另一套资料。

工具结果属于不可信输入。提示词要求 Agent 忽略文件中的操作指令和提示词注入，只提取业务相关内容。

私有文件工具可用于：

- Agent 对话发言。
- 对应 Agent 的增量记忆提取。
- 对应 Agent 的日报。

不能用于中立公共评估。

## 数据存储

### 服务器 `DATA_DIR`

服务器 SQLite 现在是工作区的唯一持续保存来源。认证后的 `GET /api/state` 恢复工作区，`PUT /api/state` 以 400ms 防抖、只提交发生变化的数据分片，并自动保存以下数据：

- 当前配置和提示词
- 投资人和创业者的多套用户资料、当前选择，以及每套资料独立的动态层、文件关联、记忆和日报
- 最多 100 个命名配置版本
- 最近 20 条模拟记录
- 模拟对话、公共结果、双方记忆、日报和调试调用
- 每个角色最多 20 个用户测试对话，以及其所属资料 ID、冻结的本 Agent、运行设置、本方文件清单、消息、工具调用、完整请求和错误；不保存或发送对方 Agent Card
- 历史模拟中的双方 Agent Card 快照
- 当前激活的配置版本和模拟记录

写请求要求有效会话并进行同源检查；单次和总工作区 JSON 上限为 24MB。保存携带服务器最近返回的 `updatedAt` 做乐观并发校验：不同页面修改不同分片时会安全合并，同一分片同时修改时返回 `409`，页面不会静默覆盖，并提示用户重新加载。保存尚未完成或失败时刷新/关闭会触发未保存保护，主动退出也会先等待并刷新保存。当前仍是单一共享工作区，不是多租户数据库：使用同一部署和账号的人会看到同一份配置与历史。

每条历史模拟记录保留当时的资料、对话、调试调用和结果快照；点击“查看”不会用历史快照覆盖当前 Agent 的长期记忆或当前日报。需要核对历史时的完整快照可直接导出该记录 JSON。

旧版本曾使用以下 `localStorage` 键：

```text
vc-agent-debugger:config:v1
vc-agent-debugger:versions:v1
vc-agent-debugger:records:v1
```

当服务器还没有工作区记录时，页面会读取一次这些旧键、逐条校验版本与模拟记录、迁移可用数据到服务器，并对损坏条目给出警告；迁移后删除浏览器副本，之后不再持续写 `localStorage`。如果服务器已有数据，则服务器数据优先，旧浏览器键会被清理。

默认目录为项目下的 `data/`：

```text
data/
├── app.db
└── uploads/
    ├── investor/<file-id>/source.ext
    └── founder/<file-id>/source.ext
```

SQLite 中保存：

- 完整工作区状态
- 文件元数据
- 文件文本分块
- 登录失败限流记录

`data/` 和 `.env` 均被 Git 忽略。Docker Compose 使用持久化卷保存 `/app/data`。

当前服务器留存仍有明确限制：没有账号/租户级所有权、字段级权限、自动过期、删除审计、静态加密或内置备份。用于真实资料前应先补齐这些能力，并确认所选模型供应商对请求内容的处理与留存政策。

## 安全边界

- API Key 仅由服务端读取。
- 登录使用服务端校验和 HMAC 签名的 HttpOnly Cookie。
- Cookie 使用 `SameSite=Strict`，生产环境增加 `Secure`。
- 登录失败按用户名和客户端来源限流，记录持久化在 SQLite。
- 写请求进行同源检查。
- 文件 API 和模型 API 都要求有效会话。
- 文件名经过处理，上传格式、大小和数量受限。
- Agent 文件作用域在服务端强制隔离。
- CSP 禁止第三方脚本、页面嵌入、对象资源、摄像头、麦克风和定位权限。仅 `next dev` 增加 React 开发调试需要的 `'unsafe-eval'`；生产构建仍禁止 `eval`。
- `test` 只是本地演示密码，公网部署前必须修改。
- 当前只有单一共享账号和单一服务器工作区，不是正式的多用户授权、租户隔离或数据权限系统。

| `APP_ORIGIN` | 公网建议 | 唯一外部站点地址，用于严格同源校验 |
| `TRUST_PROXY` | 否 | 仅在可信反向代理清洗转发头时设为 `1` |

不同 OpenAI 兼容服务价格不同，因此模型单价不通过环境变量写死，而在页面工作区中配置并随服务器状态保存。

## Docker 部署

部署前必须：

1. 修改 `AUTH_PASSWORD`。
2. 设置至少 32 字符的随机 `AUTH_SESSION_SECRET`。
3. 设置正确的 `APP_ORIGIN`。
4. 配置模型 API 参数。

启动：

```bash
docker compose up -d --build
```

Compose 默认只将端口绑定到 `127.0.0.1:3000`，应在同机使用 Nginx 或 Caddy 提供 HTTPS 反向代理。

持久化卷：

```text
venture-agent-data → /app/data
```

上线试用前还应考虑：

- IP 白名单或 VPN
- 反向代理全局限流
- 请求体大小限制
- HSTS
- 数据卷备份
- 定期轮换密码和会话密钥
